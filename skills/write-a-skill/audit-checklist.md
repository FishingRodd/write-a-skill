# Skill Audit Checklist

AUDIT 模式下按此清单逐维度诊断。每次只处理一个小批次维度：优先 2-3 个维度，诊断完必须停下等待用户确认是否继续或修改。

---

## 维度 1：Description 字段质量

**检查项**：
- [ ] 长度 ≤ 1024 字符
- [ ] 含 "Use when" + 具体触发词/场景（非泛化描述）
- [ ] 含 "Do NOT use for" + 明确排除场景
- [ ] 第一句说明 capability（第三人称）

**常见问题**：
- "Use when user needs help with X" → 太泛，agent 无法判断何时触发
- 无 "Do NOT use for" → skill 与其他 skill 边界模糊，容易误触发
- 超过 1024 字符 → 超出 agent 读取限制，可能被截断

**诊断问法**：description 的触发词够不够具体？能否区分于功能相似的其他 skill？

---

## 维度 2：Safety Rails（NEVER / ALWAYS）

**检查项**：
- [ ] 存在 NEVER 列表（至少 1 条）
- [ ] NEVER 条目是操作级别（动词开头，不是"小心X"而是"不做X"）
- [ ] 存在 ALWAYS 列表（如有强制操作）
- [ ] 高风险操作（删除/覆盖/推送）均有对应 NEVER 或 ASK 保护

**常见问题**：
- 只有"注意事项"文字，没有格式化的 NEVER 列表 → agent 不会当作硬约束执行
- NEVER 条目太抽象（"不要犯错"）→ 无法机械执行

**诊断问法**：这个 skill 最灾难性的误操作是什么？是否在 NEVER 里明确列出？

---

## 维度 3：Phase 拆分与完成标准

**检查项**：
- [ ] 复杂任务已拆分为明确的 Phase（防止目标偏移）
- [ ] 每个 Phase 有可测量的完成标准
- [ ] 复杂 skill 的关键 Phase 有 Governance Gate（进入/退出条件、阻塞条件或验证方式）
- [ ] Phase 之间的依赖关系清晰（顺序执行 vs 可并行）
- [ ] 没有"大而全"的单一阶段（信号：某 Phase 超 20 个步骤）

**常见问题**：
- 线性步骤列表替代 Phase 结构 → agent 在长任务中容易偏移目标
- 完成标准是主观描述（"完成后"）→ agent 无法判断何时推进

**诊断问法**：最长的一条执行路径有多少步骤？是否需要拆分成子目标？

---

## 维度 4：BLOCK / ASK / LOG 决策表

**检查项**：
- [ ] 存在决策表（三类决策各至少 1 条）
- [ ] BLOCK 条件明确（什么情况必须停止）
- [ ] ASK 包含推荐答案（不只是"等用户说"）
- [ ] LOG 有回报机制（自主处理后告知用户）

**常见问题**：
- 只有 BLOCK，没有 ASK/LOG → agent 过于保守，频繁打断用户
- ASK 没有推荐答案 → 给用户增加决策负担
- 所有情况都是 ASK → 失去自主执行能力

**诊断问法**：这个 skill 的最高风险边界情况是什么？agent 遇到时应该停下来还是自主处理？

---

## 维度 5：规则晋升机制

**检查项**：
- [ ] 有 `[OBSERVE]` 标记说明
- [ ] 有 LOG→ASK→AUTO 的晋升路径说明
- [ ] 有降级路径（AUTO 误判 → 回退）
- [ ] 有 neat-freak 后处理触发说明

**适用场景**：频繁调用、边界复杂、或会持续积累案例的 skill。
**可跳过**：偶发性 skill（如一次性交接、纯文本压缩）。

**诊断问法**：这个 skill 是否会被高频调用并积累边界情况？是否值得投入规则晋升机制？

---

## 维度 6：Progressive Disclosure（文件分档）

**检查项**：
- [ ] SKILL.md 不超过 100 行（或超出时已拆出支撑文件）
- [ ] 支撑文件有"读取时机"说明（按需加载）
- [ ] 每条治理规则有唯一权威文件，其他文件只引用、展开或举例
- [ ] 模板没有成为隐藏规范源（模板中的强约束已回写权威文件）
- [ ] pitfalls.md 存在（哪怕为空）
- [ ] 无大量复制粘贴代码（重复 ≥ 3 次 → 提取到 examples.md）

**常见问题**：
- SKILL.md 超过 300 行 → agent 每次加载消耗大量 token，且关键信息被稀释
- 无读取时机说明 → agent 会一次性读所有支撑文件（token 浪费）

**诊断问法**：哪些内容是每次必须加载的？哪些只在特定情况才需要？

---

## 维度 7：长任务分解（防目标偏移）

**检查项**：
- [ ] 超过 5 步的操作序列已拆分为带检查点的 Phase
- [ ] 每个检查点有明确的验证操作（不只是"看看有没有问题"）
- [ ] 存在"子任务完成后汇报"机制（让用户知道进度）

**常见问题**：
- Agent 在长任务中途改变目标（"顺便修了其他 bug"）→ 缺少 Phase 边界约束
- 一次性执行大量操作无中间确认 → 出错后难以回滚

**诊断问法**：这个 skill 最长的执行序列是什么？中途是否有自然检查点可以插入验证？

---

## 维度 8：知识沉淀路径

**检查项**：
- [ ] pitfalls.md 存在且格式一致（症状 → 原因 → 修复）
- [ ] 新经验已分类为 PATTERN / RULE / TEMPLATE / EVAL / PITFALL
- [ ] 一次性 Task Context 未直接晋升为 Core Rule
- [ ] 有从会话经验到 pitfalls.md 的写入机制（如 [OBSERVE] 或 neat-freak）
- [ ] pitfalls 按 domain 分类（不是时间顺序）
- [ ] 过期/已修复的 pitfall 有清理机制

**常见问题**：
- pitfalls 只记录"遇到过"，不记录修复方法 → 下次还是要重新排查
- 按时间顺序记录 → 难以检索
- 无清理机制 → 积累大量过时信息

**诊断问法**：这个 skill 最典型的踩坑场景是什么？是否已有 pitfalls.md 记录并提供解决方案？

---

## 维度 9：外部设计学习能力

**检查项**：
- [ ] 支持从文章、agent session、其他 skill 中提炼可复用设计
- [ ] 明确“只吸收有证据支撑的内容”，不可访问来源不得臆造
- [ ] 有归档分类：description / workflow / decision / context / memory / anti-pattern
- [ ] 每个吸收点说明：来源证据 → 机制类型 → 写入位置 → 验证方式
- [ ] 只吸收能改变 skill 行为的机制，不只沉淀理念或口号
- [ ] 来源证据不足的内容没有写成 RULE
- [ ] 能判断写入目标：SKILL.md / audit-checklist.md / templates / patterns.md / pitfalls.md

**常见问题**：
- 文章不可访问时仍编造总结 → 污染 skill 规则
- 只做阅读总结，不转化为可执行规则或模板
- 看到优秀设计只评价，不吸收进自身方法论

**诊断问法**：这个 skill 是否能持续从外部优秀设计中学习？学习结果是否能落到具体文件和规则？

---

## 维度 10：Harness 级治理（复杂 / 多 Agent / MCP / 高风险工具）

**检查项**：
- [ ] 明确“Agent 负责局部智能，Skill 负责全局约束”
- [ ] 有任务生命周期：创建 → 规划 → 执行 → 验证 → 完成 / 失败
- [ ] 有硬终止条件：max_steps / max_tokens / max_duration / max_tool_calls（按场景选用）
- [ ] 工具调用有治理：允许工具、风险等级、是否需要人工确认、输出结构、审计策略
- [ ] 高风险工具（文件写入、删除、代码执行、数据库写、外部系统写）有 ASK/BLOCK 规则
- [ ] 有 trace / execution log，用于回答“为什么这个结果可信”

**适用场景**：多 agent、MCP、后台监控、高风险工具、长流程自动化。
**可跳过**：纯文本转换、一次性小工具、无外部副作用的简单 skill。

**常见问题**：
- 让 Planner/Agent 自己决定是否继续、重试、结束 → 全局控制权错位
- MCP Server 或工具直接暴露给 Agent → 缺少安全网关
- 没有 hard stop → Agent 循环调用工具或无限重试
- 没有执行日志 → 结果看似正确但过程不可审计

**诊断问法**：这个 skill 是否已经定义了生命周期、工具边界、预算/终止条件和 trace？

---

## 维度 11：状态、记忆与上下文预算

**检查项**：
- [ ] 区分 Core Context（跨 skill 通用规则）
- [ ] 区分 Domain Context（某类 skill 的领域约束）
- [ ] 区分 Task Context（当前一次性目标，不直接沉淀为长期规则）
- [ ] 区分 Working State（当前步骤临时上下文）
- [ ] 区分 Execution Log（不可变日志，审计/回放用）
- [ ] 区分长期 Memory（跨任务复用的经验/偏好/规则）
- [ ] 明确哪些信息不能写入长期记忆（临时状态、过期路径、一次性输出）
- [ ] 长上下文有压缩策略：保留关键事实、决策、未解决问题、工具结果引用
- [ ] 事实型/合规型任务保留原始引用，不只保留摘要

**常见问题**：
- 临时状态污染长期 memory
- 所有历史都塞进上下文，导致 token 膨胀
- 压缩丢失关键证据，后续无法追溯

**诊断问法**：这个 skill 需要记住什么、忘掉什么、压缩什么、保留什么原始证据？

---

## 维度 12：轨迹评估与回归验证

**检查项**：
- [ ] 不只验证最终输出，也验证执行轨迹
- [ ] 能追溯：目标 → 设计决策 → 文件改动 → 验证结果
- [ ] 检查是否重复调用工具、无意义重试、顺序不合理、目标偏移
- [ ] 优先使用确定性检查：schema、单元测试、命令结果、规则引擎
- [ ] LLM-as-Judge 只用于开放式表达、完整性、连贯性等非确定性评估
- [ ] 修改 prompt / skill / 工具 schema 后有回归验证方式

**常见问题**：
- 最终答案正确，但过程中调用了未授权工具
- 成功依赖偶然重试，不具备稳定性
- 每次优化靠感觉，没有回归基线

**诊断问法**：这个 skill 如何证明“过程正确”，而不只是“最后看起来对”？

---

## 维度 13：可训练性与 EVOLVE 协议

**检查项**：
- [ ] 有 GT / 测试集 / 评测标准，或有生成 GT 的流程
- [ ] GT 分为 dev / holdout / regression，holdout 不参与优化
- [ ] 每轮只做一个原子改动，能用“和”描述的改动必须拆轮
- [ ] 有三层评测：L1 Quick Gate、L2 Dev Eval、L3 Strict Eval
- [ ] 有 5 维 AND 门控：Correctness、Regression、Safety、Cost、Atomicity
- [ ] 有 per-case trace，失败诊断必须引用原始 trace 证据
- [ ] 有 keep/discard、checkpoint、rollback 机制
- [ ] 有不可修 case 标记机制，避免为错误 GT 过拟合

**适用场景**：需要持续优化命中率、召回率、通过率、成本、稳定性的 skill。
**可跳过**：一次性生成、无可量化目标、无测试样本的轻量 skill。

**常见问题**：
- 只手工试几个 case，质量取决于作者直觉
- 没有 holdout，优化器背 dev 答案导致过拟合
- LLM 评测有噪声却只跑一次，误判改动效果
- 没有 rollback，坏改动污染主 skill
- 为了涨分硬编码 case，破坏泛化能力

**诊断问法**：这个 skill 是否有可度量目标、GT、trace、门控和回滚？是否能稳定选出最佳 checkpoint？
